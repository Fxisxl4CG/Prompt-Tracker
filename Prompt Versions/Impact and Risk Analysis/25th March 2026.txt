# PASS 1 — Impact Reasoning

### Determine Initial Change Targets

Use the **Salesforce Domain Map** to identify the initial components affected by the change.

These may include:

- objects and fields  
- automation artefacts (Flows, Approval Processes)  
- Apex classes and triggers  
- integrations  
- reports and dashboards  
- security artefacts (Profiles, Permission Sets, FLS)  
- UI components  

---

### Trace Dependency Propagation

Use the **Dependency Graph Report** to determine how the change propagates.

Trace dependencies through:

- Apex call chains  
- Flow trigger chains  
- object-field relationships  
- integration callouts  
- shared utility classes  
- report types  
- permission sets  
- platform events  

---

### Classify Components by Impact Category

- Direct Impact — Modified  
- Direct Impact — Schema Dependency  
- Direct Impact — Behavioural Dependency  
- Indirect Impact — Cascade  
- Indirect Impact — Shared Component  
- Potential Impact — Boundary  
- No Impact — Confirmed  
- No Impact — Orphaned  

---

### Determine Impact Type

- Breaking  
- Behavioural Change  
- Data Impact  
- Security Impact  
- Performance Impact  
- Test Impact  
- Deployment Impact  
- UI Impact  

---

### Assign Impact Severity

- Critical  
- High  
- Medium  
- Low  
- Informational  

Severity should consider:

- dependency breadth  
- platform layer involved  
- architectural centrality of the component  

---

### Assign Confidence Level

- High  
- Medium  
- Low  
- Assumed  

---

### Determine Blast Radius

Calculate:

- directly impacted components  
- indirectly impacted components  
- boundary dependencies  

Blast radius categories:

- Contained (≤3 direct)  
- Moderate (4–10 direct)  
- Wide (11–20 direct)  
- Org-Wide (21+)

Also identify the **highest severity tightly coupled cluster**.

---

# PASS 2 — Risk Analysis

### Governor Limit Risks

Assess exposure to:

- SOQL limits  
- DML limits  
- Apex CPU time  
- Flow transaction limits  
- heap size limits  
- callout limits  

---

### Data Integrity Risks

Evaluate:

- field removals or type changes  
- required field additions  
- roll-up summary recalculation  
- cascade deletes  
- duplicate rule interactions  
- data migration ordering  

---

### Regression Risks

Identify risks affecting:

- Apex test classes  
- Flow automation logic  
- integration contracts  
- permission configurations  
- user interface behaviour  

---

### Security and Compliance Risks

Evaluate risks affecting:

- Field Level Security  
- Object access  
- Apex execution context  
- encryption policies  
- regulatory requirements  

---

### Platform and Deployment Edge Cases

Assess:

- deployment order dependencies  
- metadata API limitations  
- sandbox vs production differences  
- Salesforce seasonal release changes  
- managed package upgrade conflicts  
- recursive automation chains
